**TL;DR**: Stop treating the Model Context Protocol as a plugin format. Model each external system — every API, database, and SaaS — as its own MCP server, and you turn integrations into isolation boundaries: a failing tool returns a contained error instead of cascading through your agent. MCP is the universal port between a model and the world; A2A is the protocol between agents. Under Linux Foundation governance as of mid-2026, MCP is a stable architectural primitive you can design around.

---

## The reframe

Most teams first meet MCP as "the thing that lets Claude use my tools." True, and a wild undersell. Once you connect more than one or two tools, MCP stops being a feature and starts being the **shape of your system**.

Here is the load-bearing idea. If retrieval architecture is about *where the knowledge lives* — and if [GraphRAG is the map](/blog/agentic-rag-vs-graphrag-2026) of how entities connect — then **MCP is the universal port**: the single, typed socket through which an agent reaches every external system, regardless of what is behind it. A map tells you the territory. A port lets you dock at it. You need both.

The architectural payoff shows up under failure.

## Why one server per system

The lazy design is a monolithic tool layer: one process that wraps the billing API, the search index, the CRM, and the file store. It works in the demo. In production, when the CRM's auth token expires, the whole tool layer wobbles and your agent loses access to billing too — a failure in one integration cascading into unrelated ones.

The architectural design is **one MCP server per external system**. Billing is a server. Search is a server. The CRM is a server. Now:

- A failure in one is a **contained error** the agent receives and can route around — "search is down, answer from cache and flag it" — instead of a cascade.
- Each server is **versioned, secured, scaled, and deployed independently**. The billing team owns the billing server; you do not redeploy your agent to ship a billing change.
- The agent's tool surface is **uniform**. It does not care that one server fronts a REST API and another fronts a SQL database — both speak MCP.

This is the same reasoning that made microservices and the bulkhead pattern worth their cost: isolate the blast radius. MCP gives you the isolation boundary for free because each server is already a separate process speaking a standard protocol.

```ts
// A minimal MCP server fronting one external system: a contract store.
// One server = one system = one isolation boundary.
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'
import { z } from 'zod'
import { contractDb } from './contract-db'

const server = new McpServer({
  name: 'contracts',
  version: '1.0.0',
})

server.registerTool(
  'find_contract',
  {
    title: 'Find contract',
    description: 'Look up a contract by counterparty name.',
    inputSchema: { counterparty: z.string() },
  },
  async ({ counterparty }) => {
    try {
      const rows = await contractDb.search(counterparty)
      return {
        content: [{ type: 'text', text: JSON.stringify(rows) }],
      }
    } catch (err) {
      // Failure stays inside this server. The agent gets a clean error,
      // not a crash that takes down every other tool.
      return {
        isError: true,
        content: [{ type: 'text', text: `contracts unavailable: ${String(err)}` }],
      }
    }
  },
)

await server.connect(new StdioServerTransport())
```

The `isError` path is the whole point. When the contract store is unreachable, the agent receives a structured error scoped to *this* tool and keeps functioning. Build every server this way and graceful degradation becomes the default rather than something you bolt on.

## MCP vs A2A: two protocols, two jobs

The common confusion in 2026 is treating MCP and Agent2Agent (A2A) as competitors. They are not. They sit at different seams.

| Concern | MCP | A2A |
| --- | --- | --- |
| **Connects** | An agent to tools and data | An agent to another agent |
| **Question it answers** | "What can this agent *do*?" | "Which agent should *handle* this?" |
| **Unit behind the port** | An API, DB, or service | An autonomous agent with its own tools |
| **Typical use** | Give a single agent capabilities | Delegate, coordinate, hand off across agents |

A single agent uses **MCP** to reach its tools. A *system* of agents uses **A2A** to divide labor — a planner agent delegating a research subtask to a research agent that has its own MCP servers underneath. Most production systems use both: A2A for the org chart of agents, MCP for each agent's hands.

If you only have one agent, you only need MCP. Reach for A2A when you have genuinely separate agents with distinct responsibilities that must coordinate — not before.

## The cloud runtime landscape

Architecting around MCP means knowing how each runtime exposes it. Here is the picture as of mid-2026 — verify the current GA/preview status for your specific dependency before you commit, because this surface moves quarter to quarter.

| Runtime | Status | Tool / MCP story | Memory note |
| --- | --- | --- | --- |
| **AWS Bedrock AgentCore** | GA | Managed agent runtime with MCP-compatible tool integration; AgentCore Gateway exposes tools to agents | AgentCore Memory is GA (short + long-term) |
| **Microsoft Foundry Agent Service** (formerly Azure AI Foundry) | GA | First-class tool and MCP integration in the Foundry Agent Service | **Memory in Foundry Agent Service is preview, not GA** — do not assume Bedrock-parity managed memory |
| **Google Vertex AI (ADK)** | GA (ADK) | Agent Development Kit exposes tools, including MCP, through its tool interface | Agent Engine Sessions / Memory Bank for state |
| **Oracle OCI Enterprise AI Agents** | GA | OpenAI Responses-compatible API surface; tool calling via that compatibility layer | Pairs with Database 26ai AI Vector Search for retrieval/memory |

Two things to take from this table. First, **GA is not uniform** — Azure's agent service is GA while its memory layer is still preview, so a design that leans on managed durable memory needs a different plan on Azure than on Bedrock. Second, Oracle's choice to expose an **OpenAI Responses-compatible API** means tool-calling code written against that shape ports across with minimal change — a portability win if you are multi-cloud.

(Independent note: this is an analysis of public capabilities, not an Oracle position. This site is not affiliated with or endorsed by Oracle.)

## Governance: why this is safe to build on

A protocol you architect around needs to outlive any single vendor. As of mid-2026 MCP sits under **Linux Foundation governance** (through the Agentic AI Foundation, which took stewardship in December 2025), with a vendor-contributed enterprise authorization extension covering identity-provider-managed access and approval policy; observability standardization is on the 2026 roadmap.

That governance is the difference between adopting a clever convention and adopting a standard. It is why "build the MCP server once, use it from any agent framework" is a real architectural guarantee rather than a hope — the same server in the code sketch above works whether the agent runs on Bedrock, Foundry, Vertex, OCI, or a local Vercel AI SDK process.

## How to architect around it

A short checklist for treating MCP as the foundation it is:

1. **One server per external system.** Resist the monolithic tool layer. Each integration gets its own isolation boundary.
2. **Make every tool fail closed and contained.** Return `isError` with a scoped message; never let one tool's failure surface as a crash.
3. **Keep business logic outside the model.** The MCP server validates, authorizes, and executes; the model only plans. (Same golden rule as guardrails — logic outside the LLM.)
4. **Reserve A2A for real multi-agent systems.** Do not add an agent-to-agent protocol to a single-agent design.
5. **Pin to GA where you depend on it.** If you need durable managed memory, confirm it is GA on your runtime — on Azure as of mid-2026, it is not.

Do these and MCP stops being a curiosity in your architecture diagram and becomes the spine of it — the typed port every capability docks into, with failures held where they happen.

## FAQ

### What is the difference between MCP and A2A?

MCP (Model Context Protocol) connects one agent to tools and data sources — it is the port between a model and the systems it acts on. A2A (Agent2Agent) connects agents to each other — it is the protocol for one agent to delegate to and coordinate with another. You use MCP to give an agent capabilities and A2A to let agents cooperate.

### Why model each external system as its own MCP server?

Isolation. When the billing API and the search index are separate MCP servers, a failure or slowdown in one returns a contained error the agent can handle, instead of taking down a monolithic tool layer. It also lets you version, secure, and scale each integration independently.

### Is MCP production-ready in 2026?

As of mid-2026, MCP is under Linux Foundation governance and is supported as a first-class integration by the major cloud agent runtimes and several SDKs including the Vercel AI SDK. It has a stable transport story and a vendor-contributed enterprise authorization extension; observability standardization is on the 2026 roadmap. It is a reasonable foundation to build on, with the usual caveat to verify each runtime's current support level yourself.

### Do all cloud agent runtimes support MCP the same way?

No. Support varies. AWS Bedrock AgentCore and Oracle OCI Enterprise AI Agents are GA, Microsoft Foundry Agent Service (formerly Azure AI Foundry) is GA while its Memory layer is still preview, and Google's Vertex ADK exposes MCP through its tool interface. Always confirm the current GA/preview status of the specific runtime and feature you depend on.
