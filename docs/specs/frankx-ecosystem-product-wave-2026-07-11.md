# FrankX Ecosystem Product Wave

Date: 2026-07-11
Surface: `/ecosystem`, `/ecosystem/agentic-student-os`, `/ecosystem/agentic-life-os`
Brand: FrankX
Asset tier: C - exact interactive system interface. No external or generated media.

## Product Thesis

The public ecosystem is a decision surface, not a dump of the private repo estate. It helps a capable builder identify the operating system that matches the work, inspect the evidence behind it, and leave with a concrete first-week plan.

Private memory, family records, health data, financial records, credentials, local paths, internal commands, and unfinished commercial systems remain outside the public graph.

## ICP

Primary users are technically fluent founder-operators, creators, researchers, and advanced students who already use frontier AI tools but lack a durable operating model.

They want:

- a system they can inspect, adapt, or implement;
- a first useful result quickly;
- explicit boundaries around privacy and high-stakes decisions;
- proof in the form of a public route, repository, workflow, or generated starter plan.

They distrust:

- vague autonomous-agent claims;
- giant undifferentiated capability lists;
- fake dashboards and decorative graphs;
- products that hide whether they are open source, a blueprint, or a private engagement.

## Language Brief

Use: operating system, workflow, review gate, source, proof, private boundary, starter plan, implementation.

Avoid: revolutionize, unlock, limitless, relentless, peak performance, entire life, autonomous decisions, clinical intelligence, wealth compounding, and internal estate vocabulary.

Emotional movement: overwhelmed by scattered AI use -> oriented -> able to choose a bounded first system.

Primary action: choose the work to make operational.

## Product Loop

Input -> select a desired outcome.

Interpretation -> match the outcome to a public system and its maturity.

Inspectable result -> show audience, mechanism, first win, proof, and boundary.

Refinement -> compare adjacent systems or change the selected outcome.

Action -> open the blueprint/repository or copy a generated first-week plan.

## Scene Brief

### Scene 1 - Orientation

Job: explain that FrankX contains several bounded operating systems, not one universal product.

Composition: editorial claim, restrained supporting copy, three trust markers. No hero card, glow orb, or decorative 3D.

### Scene 2 - Navigator

Job: turn visitor intent into a recommended starting system.

Interaction: segmented outcome choices update one stable result panel. State changes the recommendation, proof route, boundary, and first win.

Mobile: choices become a two-column control grid; result remains below the controls.

Reduced motion: no dependency on motion; state swaps are immediate.

### Scene 3 - Public System Families

Job: make the wider portfolio inspectable without exposing the private estate.

Composition: concise repeated system records grouped by customer outcome and maturity.

### Scene 4 - Relationship Map

Job: explain how public systems connect to the Starlight substrate.

WebGL decision: retain the existing map only as an optional secondary lens. It is not the hero or product proof. It must use the public-safe registry, lazy-load on desktop, and preserve static mobile/reduced-motion fallbacks.

### Scene 5 - Boundary And Next Action

Job: state what is public, private, and human-controlled; route to the appropriate blueprint or implementation conversation.

## Release Gates

- Public graph contains no internal-only registry entries.
- Student OS is labeled as a public blueprint, not a shipped software product.
- Life OS is labeled as a private operator suite with a public blueprint.
- Every recommended system has a real proof route or repository.
- Copy audit, TypeScript, scoped lint, desktop/mobile/reduced-motion inspection, and Vercel preview are required before production consideration.
