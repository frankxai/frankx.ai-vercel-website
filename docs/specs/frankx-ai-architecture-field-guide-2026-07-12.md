# FrankX AI Architecture Field Guide

## Outcome contract

- Audience: AI architects, technical founders, and platform leads choosing a deployable agent-system baseline.
- Surface: public `/ai-architecture` authority and reference hub.
- First read: build an agent system you can operate, using Vercel for the experience, Railway for persistent work, and GCP for managed intelligence where justified.
- First win: filter official architectures by deployment target and open maintained documentation plus a working repository in under one minute.
- Product loop: choose target → compare architecture flow → inspect official docs → open working repository → adapt through FrankX blueprints.
- Proof: source-owned documentation and repositories, HTTP verification, explicit request flows, and local route contracts.
- Boundary: this page recommends integration boundaries; it does not claim that one stack fits every workload or that linked repositories can be deployed unchanged.

## Direction board

- Keep: obsidian base, editorial type, direct technical language, existing rounded primary actions.
- Avoid: ambient orbs as the subject, fake dashboards, generic agent copy, unsupported cost numbers, repeated entrance animations, all-caps interface copy.
- Signature: one inspectable three-stage topology that assigns a clear job to Vercel, Railway, and GCP.
- Asset: Tier C code-authored system diagram. Exact labels and links make code the correct medium.
- Motion: local filter state only. The architecture must remain fully legible without animation.

## Scene brief

1. Hook: outcome-led headline beside the three-stage deployment topology.
2. Role split: concise explanation of what each platform owns.
3. Source atlas: filterable official architecture rows with flow, docs, repository, and deployment targets.
4. Adaptation: route into FrankX implementation blueprints.

## Verification plan

- Lint and typecheck the scoped implementation.
- Run the local architecture contract.
- Fetch every linked source and follow redirects; require HTTP 2xx.
- Deploy one Vercel preview from the owned feature branch.
- Inspect desktop, mobile, filter state, keyboard focus, and linked routes when machine/browser admission allows.
- Keep production promotion human-gated.
