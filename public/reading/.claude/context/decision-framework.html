Decision Framework for FrankX Projects

Load this context before making structural decisions.

The Prime Directive

Don't optimize for impressive-sounding metrics. Optimize for actual outcomes.

  • "60% route reduction" sounds good but may cause SEO damage
  • "Cleaner architecture" sounds good but may confuse users
  • "Consolidated structure" sounds good but may lose functionality

Ask: Does this change HELP USERS or just LOOK GOOD?

Pre-Action Checklist

Before proposing changes to:

  • URLs/routes
  • Page structure
  • Navigation
  • Information architecture
  • File organization

Complete this checklist:

## Decision Checklist

### Problem Definition
- [ ] What specific problem are we solving?
- [ ] Who experiences this problem? (Users? Frank? SEO? Maintenance?)
- [ ] What evidence do we have? (Analytics? Feedback? Errors?)
- [ ] Is this a real problem or an "architecture smell"?

### Solution Evaluation
- [ ] What's the SIMPLEST solution?
- [ ] Can we fix config instead of restructuring?
- [ ] Can we hide instead of delete?
- [ ] Can we redirect instead of rename?
- [ ] What are we optimizing for? (Users? Metrics? Aesthetics?)

### Risk Assessment
- [ ] What could go wrong?
- [ ] Is this reversible?
- [ ] SEO impact? (URL changes lose link equity)
- [ ] User confusion? (Existing bookmarks break)
- [ ] Data loss? (Deleted content is gone)

### Approval Check
- [ ] If irreversible: Got explicit approval?
- [ ] If URL change: Justified the SEO cost?
- [ ] If deletion: Verified no traffic/backlinks?

Common Mistakes to Avoid

Mistake 1: Consolidation Theater

Bad: "Let's reduce 127 routes to 50 for cleaner architecture" Why it's bad: Routes ≠ navigation. Having pages doesn't hurt. Deleting them does. Good: "Let's fix navigation to show 6 clear options. Pages stay, nav gets cleaner."

Mistake 2: URL Vanity

Bad: "Let's rename /ai-architect to /architect - shorter is better" Why it's bad: "AI Architect" is the brand. Shorter loses meaning. URL change hurts SEO. Good: Keep /ai-architect. It's established, has backlinks, and is the brand term.

Mistake 3: Spec Literalism

Bad: "The spec says consolidate to 6 hubs, so I'll delete 70 routes" Why it's bad: Following specs blindly without questioning if they make sense Good: "The spec says 6 hubs. That probably means 6 nav items, not 6 total pages."

Mistake 4: Impressive Metrics

Bad: "We achieved 60% route reduction!" Why it's bad: Optimizing for the metric, not the outcome Good: "Users can now find content in 2 clicks instead of 5" (actual improvement)

Mistake 5: Premature Optimization

Bad: "Let's reorganize everything before we have users" Why it's bad: You don't know what the right structure is yet Good: "Let's ship, see how people use it, then optimize based on data"

FrankX-Specific Rules

URLs That NEVER Change

  • /ai-architect - Brand identity term
  • /soulbook - Established lead magnet
  • /blog/* - SEO value in existing posts
  • /products/* - May have backlinks

Things That CAN Be Simplified

  • Navigation components (show fewer items)
  • Mega menu structure (better grouping)
  • Mobile nav (bottom bar + drawer)
  • Internal linking (smarter related content)

Red Flags to Pause On

  • "Delete" - Almost always wrong. Hide instead.
  • "Rename URL" - Almost always costs more than it gains.
  • "Move to subfolder" - Might bury important content.
  • "For consistency" - Consistency isn't worth SEO loss.

Escalation Protocol

If a proposed change:

  1. Changes any URL → Explain SEO tradeoff, get approval
  2. Deletes any page → Check analytics first, get approval
  3. Restructures navigation → Get approval on new structure
  4. Affects products/lead magnets → Always get approval

When in doubt: Ask Frank before acting.

The /superintelligence Upgrade

When using /superintelligence or deep thinking:

  1. Start with problem definition - Not solution design
  2. Challenge your first instinct - It's often wrong
  3. Ask "what could go wrong?" - Before "how do we do this?"
  4. Prefer minimal intervention - The best change is often the smallest one
  5. Optimize for outcomes - Not impressive-sounding metrics

This framework exists because Claude once proposed deleting 70+ pages and renaming established URLs for "consolidation." That would have hurt SEO, confused users, and solved no real problem. Learn from this mistake.
