Meta-Prompting (Starlight Integration)

Each team has context injection via Starlight:

  1. Identity Layer: Team persona, decision rules, quality standards
  2. Knowledge Layer: Project state, dependencies, constraints
  3. Strategy Layer: Product roadmap, user segments, KPIs
  4. Agents Layer: Available team members, escalation paths
  5. Memory Layer: Past decisions, patterns, learned optimizations

Example: Frontend Engineer Context

IDENTITY: "UI specialist obsessed with glassmorphic dark design, accessibility, 60fps animations"
KNOWLEDGE: "FrankX brand system: navy #0F172A, purple #AB47C7, cyan #43BFE3"
STRATEGY: "Every page must convert—CTAs above fold, email capture optional, premium feel"
AGENTS: "Can spawn design-reviewer, accessibility-auditor, perf-guardian"
MEMORY: "Framer Motion causes bus errors on heavy builds—use CSS animations instead"

Integration Points

With ACOS

  • Teams inherit ACOS safety (circuit breaker, audit trail, self-modify gate)
  • Share 75+ global skills via skill-rules.json
  • Escalate complex decisions to /council (multi-agent decision making)

With MCP Servers

  • v0 Design: Figma → React component → production code
  • Stripe MCP: Payment setup, webhook testing, revenue tracking
  • Resend MCP: Email delivery, template testing, analytics
  • Vercel MCP: Preview URLs, analytics, deployment tracking
  • Notion MCP: Specs, roadmaps, decision logs

With Starlight

  • Context pull (SessionStart): Load team context, project state
  • Decision logging (SessionStop): Save decisions, learnings, metrics
  • Pattern detection: Identify what works, codify into skill.md

Quality Gates

Design Gate

  • ✅ Spec complete (PRD, flows, wireframes)
  • ✅ Brand alignment check (design-lab/nature inspiration)
  • ✅ Accessibility pre-flight (WCAG 2.2)

Build Gate

  • ✅ Unit tests (>80% coverage)
  • ✅ Component isolation (Storybook/v0)
  • ✅ API integration tested

QA Gate

  • ✅ E2E test matrix passed
  • ✅ Performance baseline met (Core Web Vitals)
  • ✅ Accessibility audit passed (WCAG AAA)
  • ✅ Security scan passed (SAST)
  • ✅ UX review complete (conversion clarity)

Deploy Gate

  • ✅ All QA gates passed
  • ✅ Preview URL approved by Frank
  • ✅ Analytics configured
  • ✅ Rollback plan documented
  • ✅ Monitoring alerts set

Iteration Loop

  1. Launch → Team builds product
  2. Monitor → Collect user feedback, metrics
  3. Analyze → Starlight detects patterns (what worked, what didn't)
  4. Retro → Teams review learnings, update skill.md
  5. Codify → Successful patterns become reusable skills
  6. Repeat → Next product launch starts with better patterns

Orchestration Architecture

Approach: Native Claude Code + Adapted Patterns

No external framework dependencies. Orchestration uses Claude Code's Task tool for parallelism, with architectural patterns adapted from:

Pattern Source How We Use It
Gatekeeper Check OMO/Momus Pre-flight validation (clarity, verification, context, scope) before any build starts
Complexity Router shinpr/claude-code-workflows 1-2 files → direct, 3-5 → sprint, 6+ → full pipeline. Don't over-orchestrate
Dependency Levels claude-flow 4 levels (0→3): Architecture → Build → QA → Deploy
Parallel Spawning claude-flow All Level 1 agents launch in ONE message
Model Tiering wshobson/agents, metaswarm Opus for architecture/review, Sonnet for implementation, Haiku for validation
Todo Continuation OMO/Sisyphus Agents can't abandon; build report = state checkpoint for resume
Wisdom Accumulation OMO/Sisyphus Learnings extracted after each level, passed forward
Independent Verification OMO/Atlas, metaswarm Orchestrator NEVER trusts subagent claims — re-reads files, runs diagnostics
Category Routing OMO Tasks specify intent (frontend, backend, content) not model names
Circuit Breaker ACOS v10.1 3+ errors = pause. Failed agent = report, don't retry silently
Auto-Detection skill-rules.json UserPromptSubmit hook surfaces /product-team-launch on product dev intent

Why Not Install OMO or claude-flow Directly?

  • Zero dependencies — nothing to install, update, or break
  • Portable — skill.md files work with Claude Code, Cursor, Windsurf via Starlight
  • Cleaned up claude-flow in v10.1 — learned that external orchestration frameworks add complexity without proportional value
  • The LLM IS the orchestrator — Task tool + prompt engineering = swarm runtime
