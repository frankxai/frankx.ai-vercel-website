The Five Performance Killers

1. Swap Thrashing (Impact: CRITICAL)

  • Each page fault to swap = ~10ms vs ~100ns for RAM
  • That's a 100,000x penalty per swapped memory access
  • When 5 sessions all page simultaneously, disk becomes saturated
  • Fix: Fewer sessions, or kill idle ones

2. MCP Server Multiplication (Impact: HIGH)

  • Each Claude session spawns its OWN set of MCP servers
  • 5 sessions × 9 MCP servers = 45 Node.js processes
  • Each Node process: 30-80MB RSS
  • Total MCP overhead: 1.5-3.6GB
  • Fix: Disable unused MCP servers per project scope

3. C: Drive Full (Impact: HIGH)

  • Windows pagefile lives on C:
  • When C: < 5GB free, Windows can't expand pagefile
  • WSL virtual disk (ext4.vhdx) also lives on C:
  • Fix: Run /storage-intelligence, delete old files

4. NTFS Cross-Filesystem Penalty (Impact: MEDIUM)

  • Every /mnt/c/ file op goes through 9P protocol bridge
  • 10-50x slower than native ext4 (/home/, /tmp/)
  • Running builds, git, npm on /mnt/c/ while memory-constrained = compounded slowness
  • Fix: Clone repos to WSL native filesystem for heavy I/O

5. Orphaned Processes (Impact: MEDIUM)

  • When Claude sessions crash or are force-killed, child processes survive
  • Orphaned Node processes (parent PID = 1) hold memory indefinitely
  • Fix: Kill orphans with parent PID 1 and 0% CPU

Quick Diagnostic Commands

# How bad is it? (one-liner health check)
echo "RAM: $(free -h | awk '/Mem:/{print $3"/"$2}') | Swap: $(free -h | awk '/Swap:/{print $3"/"$2}') | C: $(df -h /mnt/c | awk 'NR==2{print $4" free"}') | Claude: $(ps aux|grep -c '[c]laude') procs | Node: $(ps aux|grep -c '[n]ode') procs"

# Which sessions are idle? (0% CPU = idle)
ps aux | grep "[c]laude" | awk '{if($3==0.0) print "IDLE:", $2, $6/1024"MB"; else print "ACTIVE:", $2, $6/1024"MB", $3"%CPU"}'

# Orphaned Node processes (safe to kill)
ps -eo pid,ppid,pcpu,rss,comm | awk '$2==1 && $5=="node" && $3==0.0 {printf "Orphan PID %s — %sMB\n", $1, $4/1024}'

# Emergency: what's eating all the memory?
ps aux --sort=-rss | head -20

Emergency Recovery Playbook

If the machine is nearly frozen:

  1. From Windows Terminal (not WSL): wsl --list --running to see running distros
  2. Kill one WSL distro: wsl -t Ubuntu-24.04 then wsl to restart fresh
  3. Nuclear option: wsl --shutdown — kills everything, clean slate
  4. After restart: Only open 2-3 Claude sessions max
  5. Prevent recurrence: Run /performance-engineer scan after opening 3rd session

.wslconfig Optimization Guide

Located at: C:\Users\Frank\.wslconfig

[wsl2]
# === Memory ===
# Rule: Leave 4GB for Windows on a 16GB machine
memory=12GB

# === Swap ===
# Rule: Match RAM limit. Prevents OOM kills but slower than RAM
swap=12GB

# === CPU ===
# Rule: Use most cores. Leave 2 for Windows responsiveness
processors=16

# === Memory Reclamation ===
# dropcache = most aggressive — returns freed pages to host immediately
# gradual = gentler but slower recovery
autoMemoryReclaim=dropcache

# === Disk ===
# Sparse VHD auto-compacts, preventing ext4.vhdx from growing forever
sparseVhd=true

[interop]
# Disable Windows PATH injection (faster shell startup, cleaner env)
enabled=false
appendWindowsPath=false

After changing .wslconfig: Must run wsl --shutdown from Windows Terminal, then restart WSL.

MCP Server Optimization

Not every project needs every MCP server. Create project-specific .mcp.json files:

Heavy project (full-stack web dev):

  • Keep: Vercel, Figma, Notion, memory
  • Drop: n8n (unless working on automations), playwright (unless testing)

Light project (writing, planning):

  • Keep: memory, Notion
  • Drop: Vercel, Figma, playwright, n8n, resend

Automation project (n8n workflows):

  • Keep: n8n, memory, Slack
  • Drop: Vercel, Figma, playwright

Each dropped MCP server saves 30-80MB per session.

Anti-Patterns

  • Don't suggest closing ALL sessions — user needs at least 1-2 to work
  • Don't suggest moving repos off C: if they're already there — too disruptive
  • Don't modify .wslconfig without asking — it requires full WSL restart
  • Don't recommend "upgrade RAM" as primary fix — optimize first
  • Don't kill processes without identifying them — always name what you're killing
