Product Sprint — Focused Build Phase

Execute a focused build sprint for an existing product. Skips architecture (spec already exists) and goes straight to build + QA.

Input

  • Target: $ARGUMENTS (page path or component to build/improve, e.g., "/vibe/producer" or "EmailSignup component")

Step 1: Understand Current State

  1. Read the target file(s) to understand what exists
  2. Read surrounding files (layout, shared components, data sources)
  3. Identify what needs to change or be built

Step 2: Build (Parallel where possible)

If changes span frontend + backend + content, launch parallel Task agents:

Frontend changes (Task, subagent_type: "general-purpose"):

  • Read .claude/agents/product-teams/frontend-engineer.md
  • Implement UI changes following glassmorphic dark design
  • Use existing component patterns from the codebase

Backend changes (Task, subagent_type: "general-purpose"):

  • Read .claude/agents/product-teams/backend-engineer.md
  • Implement API/data changes with Zod validation

Content changes (inline, no Task needed for copy tweaks):

  • Apply voice guidelines from .claude/agents/product-teams/content-strategist.md
  • Update copy, meta tags, SEO elements

If changes are small/focused, do them inline without spawning agents.

Step 3: Verify

  • Check TypeScript compiles (npx tsc --noEmit)
  • Verify page loads in dev server
  • Quick accessibility scan (headings, alt text, contrast)
  • Confirm no regressions to adjacent pages

Step 4: Report

Summarize changes made, files modified, and any issues found. Do NOT deploy — sprint is build-only. Use /product-deploy when ready.
