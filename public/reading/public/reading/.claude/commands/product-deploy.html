Product Deploy — Production Release

Deploy changes to production (frankx.ai) via the two-repo workflow.

Input

  • What: $ARGUMENTS (description of what's being deployed, e.g., "Vibe Producer page + newsletter updates")

Step 1: Pre-Deploy Verification

  1. Run npx tsc --noEmit — must pass with zero errors
  2. Run npm run build — must complete successfully
  3. Check git status — identify all changed/new files to deploy
  4. Verify no secrets or test data in changed files

If any check fails: Stop and fix before proceeding.

Step 2: Stage Changes in Dev Repo

  1. git status — show all changes
  2. git diff — review what's changing
  3. Present the change summary to Frank
  4. Ask: "These files will be deployed to production. Approve?"

Step 3: Sync to Production

Only after explicit approval:

# 1. Copy changed files to production worktree
# For each changed file, copy to .worktrees/vercel-ui-ux/

# 2. Commit in production repo
cd .worktrees/vercel-ui-ux
git add [specific files]
git commit -m "feat: [description]"

# 3. Push (use TOKEN-based push for WSL)
TOKEN=$(gh auth token) && git -c "http.https://github.com/.extraheader=Authorization: Basic $(echo -n "x-access-token:$TOKEN" | base64 --wrap=0)" -c credential.helper= push origin main

Step 4: Post-Deploy Verification

  1. Check Vercel deployment status (via Vercel MCP or dashboard)
  2. Verify the deployed page loads at frankx.ai
  3. Quick smoke test (links work, images load, no console errors)

Step 5: Report

## Deployment Report

**Status**: SUCCESS / FAILED

Rollback Plan

If something breaks after deploy:

  1. cd .worktrees/vercel-ui-ux && git revert HEAD
  2. Push revert (same TOKEN-based push)
  3. Verify frankx.ai reverts to previous version
  4. Investigate root cause in dev repo
