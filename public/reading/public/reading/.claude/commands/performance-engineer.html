/performance-engineer — WSL + Claude Code Performance Triage & Optimization

Purpose: Diagnose why this Windows/WSL2 machine is slow, identify which Claude Code sessions and MCP servers are consuming resources, and execute safe optimizations — all through a 3-lens performance council.

Power: Process-level visibility into every Claude Code instance + MCP server + Node process, memory/swap/disk triage, and actionable fixes that don't kill your work.

Input

  • Mode: $ARGUMENTS (default: full | scan | triage | nuke | session-report)
    • scan — read-only diagnostic, no actions taken
    • triage — auto-fix safe items + recommendations for the rest
    • nuke — aggressive mode: kill all non-current Claude sessions + purge caches (asks first)
    • session-report — just show what each Claude Code session is doing (process tree)
    • (no args) — full: scan + council + triage + report

PHASE 1: DIAGNOSTIC SCAN

Run ALL diagnostic commands. Collect results into internal state.

1.1 Memory & Swap Pressure

# Overall memory
free -h
# Swap breakdown
cat /proc/swaps 2>/dev/null
swapon --show 2>/dev/null
# Memory pressure (PSI — Pressure Stall Information)
cat /proc/pressure/memory 2>/dev/null
# OOM score for top consumers
ps -eo pid,oom_score,rss:10,comm --sort=-rss | head -25

Classify:

  • RAM > 85% used → YELLOW
  • RAM > 92% used → RED
  • Swap > 50% used → YELLOW (swap = disk speed, ~100x slower than RAM)
  • Swap > 75% used → RED (thrashing — this is your #1 slowdown cause)

1.2 Claude Code Process Census

# Find all Claude-related processes with memory usage
ps aux | grep -E "(claude|anthropic)" | grep -v grep

# Count Claude Code main processes (each session = 1 main process)
ps aux | grep -E "claude" | grep -v grep | wc -l

# Node.js processes (MCP servers, hooks, etc.)
ps aux | grep "[n]ode" | awk '{printf "%-8s %-6s %-6s %s\n", $1, $2, $6/1024"MB", $11" "$12" "$13}' | sort -t'M' -k3 -rn | head -30

# Process tree showing parent-child relationships
# Each Claude Code session spawns: main + MCP servers + language servers
pstree -p -a $(whoami) 2>/dev/null | head -80

Per-session accounting: Group Node processes by parent PID. Each Claude Code session typically spawns:

  • 1 main Claude process
  • 1 Node process per MCP server (n8n-mcp, memory, playwright, resend, Vercel, Figma, Slack, Zapier, Notion = up to 9+)
  • 1+ hook runner processes
  • Total per session: ~200-600MB RAM depending on MCP server count

1.3 MCP Server Inventory

# Find all running MCP-related processes
ps aux | grep -E "(mcp|supergateway|stdio)" | grep -v grep

# Check which MCP servers are configured
cat ~/.claude.json 2>/dev/null | grep -o '"[^"]*":\s*{' | head -20

# Memory per MCP server type
ps aux | grep "[n]ode" | grep -i mcp | awk '{print $6/1024"MB", $11, $12, $13}' | sort -rn

1.4 Disk I/O & Filesystem Health

# Disk space — C: drive free space is critical for Windows paging
df -h /mnt/c / 2>/dev/null

# I/O wait (high = disk is bottleneck)
iostat -x 1 2 2>/dev/null | tail -20 || cat /proc/diskstats 2>/dev/null | head -5

# Check if Windows pagefile can breathe
# Windows needs 10-15GB free on C: for pagefile + temp
# Below 5GB free = critical — Windows starts refusing allocations

Critical thresholds for C: drive:

  • < 15GB free → YELLOW (pagefile constrained)
  • < 5GB free → RED (Windows itself becomes unstable)
  • < 2GB free → EMERGENCY (risk of data loss, corruption)

1.5 WSL Configuration Audit

# Current WSL config
cat /mnt/c/Users/Frank/.wslconfig 2>/dev/null

# Actual WSL memory limit vs host
echo "WSL allocated: $(free -h | awk '/Mem:/{print $2}')"
echo "Host total: check Windows Task Manager"

# WSL kernel version (affects memory management)
uname -r

# Top Windows processes by memory (via /proc simulation)
# This helps identify if VS Code, Chrome, etc. are also competing
powershell.exe -NoProfile -Command "Get-Process | Sort-Object WorkingSet64 -Descending | Select-Object -First 15 Name, @{N='MB';E={[math]::Round(\$_.WorkingSet64/1MB)}}" 2>/dev/null

PHASE 2: PERFORMANCE COUNCIL (3-Lens Analysis)

For every finding, apply three lenses:

Lens 1: Safety Guardian

Asks: "Could this action lose in-progress work, kill an active generation, or corrupt state?"

Hard veto triggers:

  • Killing the CURRENT Claude Code session (the one running this command)
  • Killing a Claude session mid-generation (check if process CPU > 10%)
  • Deleting .claude/ config directories
  • Modifying .wslconfig without user confirmation (requires WSL restart)
  • Killing MCP servers for the current session

Lens 2: Performance Analyst

Asks: "What is the actual performance impact of this item? Quantify in MB freed or processes reduced."

High-impact items (prioritize these):

  • Swap thrashing (biggest perf killer — each swap page fault = ~10ms vs 0.0001ms for RAM)
  • Idle Claude sessions holding 200-600MB each + all their MCP servers
  • Orphaned Node processes (parent died but children remain)
  • Windows pagefile competing on a full C: drive
  • NTFS cross-filesystem operations (10-50x slower than ext4)

Low-impact items (deprioritize):

  • Small background processes (< 50MB)
  • System services
  • A single extra MCP server

Lens 3: Workflow Advocate

Asks: "Does the developer actually need this running right now?"

Active workflow indicators:

  • Claude session generated output in last 5 minutes (check process CPU/IO)
  • Session has unsaved/uncommitted work (can't kill safely)
  • MCP server was called recently (check process age vs CPU time)

Stale workflow indicators:

  • Claude session idle > 30 minutes (0% CPU, no recent I/O)
  • MCP server process running but parent Claude session is idle
  • Node process with 0% CPU and high RSS (leaked/orphaned)

Verdict Matrix

Safety Impact Workflow Verdict
SAFE HIGH STALE AUTO-FIX — kill/optimize without asking
SAFE HIGH ACTIVE RECOMMEND — explain tradeoff, ask user
SAFE LOW any SKIP — not worth the disruption
VETO any any PRESERVE — explain why
any HIGH ACTIVE-CRITICAL ESCALATE — need user decision + save work first

Known Safe Actions (skip council — always safe)

  • Drop Linux filesystem caches: echo 3 > /proc/sys/vm/drop_caches (requires sudo)
  • Kill orphaned Node processes (no parent, 0% CPU)
  • Report current state (read-only)
  • Recommend .wslconfig changes (don't apply without approval)
  • Suggest closing specific Claude sessions (don't kill them)

Known Dangerous Actions (always need approval)

  • Killing ANY Claude Code session
  • Changing .wslconfig (requires wsl --shutdown)
  • Running wsl --shutdown (kills ALL WSL sessions)
  • Clearing npm/pnpm caches (may slow next install)
  • Modifying swap settings

If mode = triage or full:

Step 1: Auto-fix safe items

# Kill orphaned Node processes (no parent Claude session)
# Only if: parent PID = 1 (orphaned) AND CPU = 0% AND age > 10min
for pid in $(ps -eo pid,ppid,pcpu,comm | awk '$2==1 && $4=="node" && $3==0.0 {print $1}'); do
  kill $pid 2>/dev/null && echo "KILLED orphan node PID $pid"
done

# Drop filesystem caches (safe, kernel reclaims as needed)
sync
echo 3 | sudo tee /proc/sys/vm/drop_caches 2>/dev/null && echo "Dropped filesystem caches"

# Clear /tmp items older than 2 days
find /tmp -maxdepth 1 -mtime +2 -not -name "." -exec rm -rf {} \; 2>/dev/null

  2. Suggest wsl --shutdown + restart if swap is still > 50%
  3. Output recovery report

WSL + Claude Code Performance Knowledge Base

Why Does It Get Slow?

  1. Swap thrashing: WSL2 uses a virtual disk for swap. When RAM is full, Linux pages memory to swap, which lives on the same physical disk as Windows. Each page fault = ~10ms vs ~100ns for RAM. With 5 Claude sessions all paging simultaneously, the disk becomes the bottleneck.

  2. MCP server multiplication: Each Claude Code session spawns its own set of MCP servers. 5 sessions × 8 MCP servers = 40 Node processes, each holding 30-80MB. That's 1.2-3.2GB just for MCP servers.

  3. C: drive full: Windows pagefile lives on C:. When C: is full, Windows can't expand the pagefile → Windows processes slow down → WSL slows down (shared kernel resources).

  4. NTFS cross-filesystem penalty: Every file operation on /mnt/c/ goes through 9P protocol (WSL↔Windows bridge). This is 10-50x slower than native ext4. Running npm install or git operations on /mnt/c/ while memory-constrained compounds the problem.

  5. No process isolation: WSL2 shares the same Linux kernel across all distributions. All Claude sessions compete for the same memory pool.

Quick Fixes (ordered by impact)

Fix Impact Risk Time
Close 2 idle Claude sessions HIGH (free ~1GB) LOW 10s
Run /storage-intelligence purge HIGH (free disk) LOW 2min
Drop filesystem caches MEDIUM (temporary) NONE 1s
Kill orphaned Node processes MEDIUM (~200MB) LOW 5s
Restart WSL (wsl --shutdown) HIGH (full reset) HIGH (kills all) 30s
Reduce MCP servers per session HIGH (long-term) LOW 5min config

.wslconfig Recommendations for Multi-Session Work

[wsl2]
# For 16GB host running 3-4 Claude sessions:
memory=12GB        # Leave 4GB for Windows
swap=12GB          # Generous swap prevents OOM kills
processors=16      # Use all logical cores
autoMemoryReclaim=dropcache  # Aggressive memory return

# For 16GB host running 1-2 Claude sessions:
memory=10GB        # More headroom for Windows
swap=8GB           # Less swap needed

Safety Rules (Non-Negotiable)

  1. Never kill the current session — the one running this command
  2. Never kill a session without user confirmation (except orphaned processes)
  3. Never modify .wslconfig without explicit approval — it requires WSL restart
  4. Never run wsl --shutdown without warning — it kills ALL WSL sessions
  5. Orphaned processes only = parent PID is 1 AND CPU is 0%
  6. Always measure before AND after — prove the improvement
  7. Recommend /storage-intelligence for disk issues — don't duplicate its work
  8. Don't kill Windows processes from WSL — only WSL-side processes
  9. Log every kill action — PID, process name, memory freed
  10. When in doubt, report and recommend — don't act
