Product QA — Quality Assurance Cycle

Run the full QA checklist from .claude/agents/product-teams/quality-assurance.md against a target page or component.

Input

  • Target: $ARGUMENTS (page URL path, e.g., "/vibe/producer" or "/products")

Step 1: Read the Target

  1. Find and read all files related to the target route
  2. Read the QA agent definition: .claude/agents/product-teams/quality-assurance.md
  3. Identify the page's components, data sources, and API calls

Step 2: Static Analysis

Run in parallel:

  • npx tsc --noEmit — TypeScript type checking
  • Review code for security issues (XSS, exposed secrets, unvalidated input)
  • Check for accessibility issues in JSX (missing alt, missing labels, heading hierarchy)
  • Check for performance anti-patterns (unnecessary "use client", missing lazy loading, oversized images)

Step 3: Quality Gate Checklist

Design Alignment

  • Colors match brand (#0F172A, #AB47C7, #43BFE3, #F59E0B, #10B981)
  • Glass opacity is 0.03 (not 0.15)
  • Typography correct (Playfair h1/h2, Inter body)
  • Icons are Heroicons (24px, 1.5px stroke)
  • Responsive at 375px, 768px, 1440px

Functional

  • Page renders without errors
  • All links resolve (no broken hrefs)
  • CTAs are functional and clear
  • Forms validate input
  • Error states handled (loading, empty, error)

Accessibility (WCAG 2.2 AAA)

  • Color contrast at least 7:1 on text
  • All images have descriptive alt text
  • Heading hierarchy correct (h1, h2, h3 in order)
  • Keyboard navigation works (no traps)
  • Focus indicators visible
  • Motion respects prefers-reduced-motion

Performance

  • No render-blocking client components at top level
  • Images use Next.js Image with sizes
  • Large components code-split where appropriate
  • No unused imports or dead code

SEO

  • Meta title present (under 60 chars)
  • Meta description present (120-160 chars)
  • OpenGraph image configured
  • 3+ internal links
  • Schema markup if applicable

Security

  • No secrets in client code
  • Inputs validated (client + server)
  • No raw HTML injection with user-provided data
  • API routes check auth where needed

Step 4: Report

Output a QA report:

## QA Report: [target]

**Status**: PASSED / FAILED (X/Y gates)

### Passed
- [check] — [detail]

### Failed
- [check] — [what's wrong] — [how to fix]

### Recommendations
- [improvement suggestions]

If critical issues found, fix them inline and re-verify.
