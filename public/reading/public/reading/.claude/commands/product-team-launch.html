Product Team Launch — Orchestrated Build

You are the Starlight Orchestrator coordinating a 6-team product build. Architecture adapted from oh-my-opencode, claude-flow, metaswarm, and shinpr/claude-code-workflows.

Input

  • Product: $ARGUMENTS (product name and brief description)

PRE-FLIGHT: Gatekeeper + Complexity Router

Before ANY work begins, run two checks:

Check 1: Gatekeeper (from OMO/Momus)

Validate against 4 criteria:

  1. Clarity: Is the brief specific enough to build? (Not "make it better")
  2. Verification: How will we know it's done? (Success criteria)
  3. Context: What already exists? (Grep the codebase FIRST)
  4. Scope: One sprint or needs decomposition?

If ANY fails: Ask Frank to clarify BEFORE proceeding.

Check 2: Adaptive Complexity Router (from shinpr + v11 SwarmOrchestrator)

Assess complexity using TWO signals — file count AND keyword scoring:

Step 1: Count expected files to create/modify.

Step 2: Keyword complexity boost — scan the brief for these complexity multipliers:

Keyword +Files
auth, login, session +2
API, endpoint, webhook +2
database, schema, migration +2
real-time, websocket, streaming +3
payment, checkout, billing +2
admin, dashboard, CRUD +3
multi-page, multi-route +2
SEO, schema markup, sitemap +1

Add keyword boosts to raw file count = adjusted complexity score.

Step 3: Route by adjusted score:

  • 1-3 adjusted → Direct topology — skip this command, implement inline (don't over-orchestrate)
  • 4-6 adjusted → Star topology — use /product-sprint (you + 2 parallel agents)
  • 7-12 adjusted → Mesh topology — continue with full /product-team-launch below (3 parallel agents)
  • 13+ adjusted → Hierarchical-Mesh — full pipeline + split Level 1 into 2 waves (dependency-ordered)

Model Tiering (from wshobson/agents + metaswarm)

  • Orchestrator (you): Opus — architecture decisions, review, quality gates
  • Build agents (Task tool): Sonnet — implementation, components, API routes
  • Quick checks (validation): Haiku — TypeScript compilation, file existence

When spawning Task agents in Level 1, use model: "sonnet" for cost efficiency.

LEVEL 0: Architecture (Sequential — must complete before Level 1)

Read .claude/agents/product-teams/product-architect.md for context.

Spec the Product

  1. Problem statement: Who has this problem? What's the evidence?
  2. Success metrics: Conversion rate, engagement, revenue target
  3. User flows: Entry → action → outcome (draw it out)
  4. Page structure: Sections, components needed, data sources
  5. Existing assets audit: What components/pages already exist that we can reuse? (Grep the codebase FIRST)

Design Alignment

  • Brand colors: navy #0F172A, purple #AB47C7, cyan #43BFE3, gold #F59E0B, emerald #10B981
  • Glassmorphic dark design (glass opacity 0.03, NOT 0.15)
  • Typography: Playfair Display (h1/h2), Inter (body), JetBrains Mono (code)
  • Icons: Heroicons only (24px, 1.5px stroke)

Output

Write a brief spec to the conversation (NOT a file). Include:

  • Sections list with component mapping
  • Data requirements (what JSON/API data needed)
  • CTA strategy (what action, where on page)
  • Reuse plan (which existing components)

GATE: Does the spec pass all 4 gatekeeper criteria? If not, refine before proceeding.

LEVEL 1: Build (Parallel — launch ALL agents in ONE message)

Launch 3 parallel Task agents simultaneously in a single response. Each agent receives:

  1. The spec from Level 0
  2. Its role-specific skill.md
  3. The learnings notepad path: .claude/plans/notepads/{product-name}/

Task 1: Frontend (subagent_type: "general-purpose")

Read `.claude/agents/product-teams/frontend-engineer.md` for quality standards.
Build the page component(s) in the appropriate `app/` route.

RULES:
- Check `components/` FIRST — reuse existing components before creating new ones
- Glassmorphic dark design, Framer Motion for entrance animations
- Responsive: 375px, 768px, 1440px
- Next.js Image with proper sizes/alt
- TypeScript strict, no `any`
- If you hit an error, log it and continue — don't silently skip

When done, write a brief summary of:
- Files created/modified (paths)
- Components reused vs new
- Any issues hit and how resolved

Task 2: Backend (subagent_type: "general-purpose")

Read `.claude/agents/product-teams/backend-engineer.md` for quality standards.
Build any needed API routes in `app/api/`.

RULES:
- Integrate with existing infrastructure (Resend for email, Lemon Squeezy for payments)
- Zod validation on all inputs
- Proper error responses (never expose internals)
- If no backend needed, report "No backend required" and exit

When done, write a brief summary of:
- API routes created (paths + methods)
- Integrations configured
- Any issues hit

Task 3: Content (subagent_type: "general-purpose")

Read `.claude/agents/product-teams/content-strategist.md` for voice guidelines.
Write all page copy — headlines, descriptions, CTAs, FAQ.

RULES:
- Voice: precise, bold, genuine. NO guru/spiritual language
- Lead with results, specific numbers (12,000+ songs, not "many")
- SEO: meta title <60 chars, description 120-160 chars
- 3+ internal links to related pages
- Schema markup if applicable (Article, FAQPage, Product)

When done, write a brief summary of:
- Copy sections written
- SEO metadata configured
- Internal links added

Wait for ALL 3 tasks to complete before Level 2.

WISDOM ACCUMULATION (adapted from OMO's Sisyphus pattern)

After Level 1 completes, before proceeding to Level 2, collect learnings:

  1. Read each Task agent's output summary
  2. Create a mental notepad of:
    • What worked: Components reused, patterns that shipped clean
    • What broke: Errors hit, workarounds needed
    • Decisions made: Why this approach over alternatives
    • Issues unresolved: Things that need follow-up

This wisdom informs the QA phase — the QA agent should know what was tricky and where to focus testing.

LEVEL 2: Quality Assurance — INDEPENDENT VERIFICATION

Critical rule (from OMO): NEVER trust subagent claims. Verify everything yourself.

Act as the QA agent. Read .claude/agents/product-teams/quality-assurance.md.

Verify the Build (don't trust — re-read)

  1. Read the actual files the build agents claim to have created/modified
  2. Run npx tsc --noEmit — must pass with zero errors
  3. Verify the page structure matches the Level 0 spec
  4. Check that reused components weren't incorrectly modified

Quality Gate Checklist

  • TypeScript compiles clean
  • Page renders (check for obvious JSX issues)
  • SEO metadata present and correct
  • Accessibility: alt text, heading hierarchy (h1→h2→h3), keyboard-navigable
  • No hardcoded data that should come from JSON/API
  • No any types, no eslint-disable comments
  • Responsive: no horizontal scroll at 375px
  • CTAs are functional and above the fold
  • Internal links resolve (not broken hrefs)

Fix and Re-verify

If issues found:

  1. Fix them inline (don't spawn another agent for small fixes)
  2. Re-run the failing check
  3. Document what was wrong and how it was fixed (this feeds the retro)

GATE: All checks pass? If not, fix and re-check. If 3+ critical issues remain after 2 fix attempts → pause and ask Frank.

LEVEL 3: Deploy Preparation (Sequential — depends on Level 2 passing)

Read .claude/agents/product-teams/deployment-lead.md.

Pre-Deploy Checklist

  • All files saved and consistent
  • TypeScript compiles (already verified in Level 2)
  • SEO metadata present
  • No hardcoded secrets or test data
  • No debug console.logs left in

Present to Frank

Ask Frank: "Ready to deploy? Here's what was built:"

  • Files created/modified (full list with paths)
  • Quality gate results (pass/fail per check)
  • Issues found and fixed during QA
  • Anything that needs Frank's decision

Do NOT push to production without explicit approval.

If Approved

Follow the two-repo deploy workflow:

Copy changed files to .worktrees/vercel-ui-ux/
Commit and push to production

CIRCUIT BREAKER RULES

  • 3+ errors in a single level → Pause and ask Frank
  • Task agent fails → Report the failure, show the error, don't retry silently
  • Spec ambiguity → Ask before building (don't guess)
  • File conflicts → Never overwrite existing working code without reading it first
  • Scope creep → If the build grows beyond the spec, stop and re-scope with Frank
