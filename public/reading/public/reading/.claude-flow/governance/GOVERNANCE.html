ACOS Governance Framework v9.2

Purpose

Prevent skill/agent bloat while enabling innovation. Every change to ACOS core systems is reviewed by 3 independent Guardians before approval. All decisions are logged transparently.

The 3 Guardians

1. Stability Guardian

Mandate: Protect proven patterns and high-success workflows.

Checks:

  • How many high-success patterns (>80%) exist?
  • Does this change affect skills/agents used in those patterns?
  • Is this a removal that could break existing workflows?

Approval logic:

  • Approve: Low risk to existing patterns
  • Caution: Change touches active skill domain, monitor after
  • Deny: Removal of component used in high-success patterns

2. Innovation Guardian

Mandate: Demand evidence that changes add measurable value.

Checks:

  • Does this change address a known skill gap (from skill-builder)?
  • Does it fill a weak spot identified by auto-enhancement?
  • Would adding this push us toward skill bloat (>30 skills)?
  • For removals: do skill suggestions conflict?

Approval logic:

  • Approve: Fills identified gap or addresses weak spot
  • Caution: Skill count approaching limit, must prove unique value
  • Deny: No evidence of improvement, duplicate functionality

3. Security Guardian

Mandate: Block changes that introduce risk.

Checks:

  • Does the change touch permission settings?
  • Does it modify governance files (requires explicit override)?
  • Do new skills match security block patterns (penetration, exploit, etc.)?
  • Does routing change bypass safety hooks?

Approval logic:

  • Approve: No security concerns
  • Caution: Touches config with permissions, needs verification
  • Deny: Matches block pattern or modifies governance directly

Decision Rules

Scenario Required Approval
Non-breaking addition 2/3 Guardians
Non-breaking modification 2/3 Guardians
Breaking removal 3/3 Guardians (unanimous)
Governance modification Explicit operator override

Override Process

When a Guardian blocks a change that the operator believes is correct:

bash .claude-flow/hooks/quality-gate.sh override "<file>" "<reason>"

Overrides are logged permanently in decisions.jsonl with timestamp and reason. The override log is never pruned — full transparency.

Decision Log

All decisions are appended to .claude-flow/governance/decisions.jsonl — one JSON object per line.

# View recent decisions
bash .claude-flow/hooks/quality-gate.sh review 20

Each decision records:

  • Timestamp
  • File modified
  • Domain affected
  • Change type (add/remove/modify)
  • Each Guardian's verdict and reasoning
  • Final approval/denial
  • Override reason (if applicable)

Evolution

This governance framework evolves with ACOS. Changes to governance itself require:

  1. Explicit operator approval (no auto-approve)
  2. Logged override with reasoning
  3. Review of impact on existing decisions
