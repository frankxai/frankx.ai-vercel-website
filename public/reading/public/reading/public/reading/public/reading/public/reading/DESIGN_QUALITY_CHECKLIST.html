FrankX Design Quality Checklist

Pre-Ship Validation for Components & Pages

Use this checklist before merging any component or page to production. All items should be checked ✅ before approval.

🎨 Visual Design

Design Tokens

  • All colors use Tailwind tokens (no hardcoded hex codes)
  • Spacing follows 8px grid (p-2, p-4, p-6, p-8, etc.)
  • Typography uses defined scale (text-heading-1, text-body, etc.)
  • Border radius matches component type (buttons: rounded-2xl, cards: rounded-3xl)
  • Shadows use defined elevation levels (shadow-lg, elevation-2, soul-glow)

Layout & Spacing

  • Micro spacing (within components): 8px, 12px, 16px
  • Component padding: 16px, 24px, 32px
  • Section spacing: 48px, 64px, 96px
  • Macro spacing (between major sections): 128px+
  • No odd spacing values (e.g., p-7 = 28px breaks grid)

Colors

  • Uses brand palette (conscious-purple, tech-cyan, music-orange, growth-green)
  • Glassmorphic backgrounds use white/5 or white/10
  • Gradient backgrounds use centralized presets (when available)
  • Semantic colors for states (success: green, warning: yellow, error: red)

Typography

  • Headings use font-bold (700) or font-extrabold (800)
  • Buttons use font-semibold (600)
  • Body text uses font-normal (400) or font-medium (500)
  • Line height appropriate (1.5× for body, 1.2× for headings)
  • No text smaller than 14px (text-sm)
  • Heading hierarchy semantic (h1 → h2 → h3, no skipping)

♿ Accessibility (WCAG AAA)

Color Contrast

  • Normal text: 7:1 minimum contrast ratio
  • Large text (18pt+ or 14pt+ bold): 4.5:1 minimum
  • UI components (borders, icons): 3:1 minimum
  • Test with WebAIM Contrast Checker
  • Gradient text tested in all lighting conditions

Touch Targets

  • All buttons minimum 44×44px (min-h-[44px])
  • Icon buttons minimum 40×40px (w-10 h-10)
  • Adequate spacing between targets (8px minimum)
  • Larger targets on mobile (48×48px recommended)

Keyboard Navigation

  • All interactive elements keyboard-accessible
  • Tab order follows logical visual flow
  • Focus indicators clearly visible (ring-2 ring-purple-500/50)
  • No keyboard traps (can navigate away from all elements)
  • Skip navigation link present (if header)
  • Escape key closes modals/menus

Screen Readers

  • Semantic HTML used (button, nav, header, main, footer)
  • ARIA labels on icon-only buttons
  • Images have descriptive alt text (or alt="" if decorative)
  • Form inputs have visible, persistent labels
  • Error messages associated with fields (aria-describedby)
  • Loading states announced (aria-live)

Forms

  • Every field has visible label (not placeholder-only)
  • Field requirements shown before submission
  • Error messages specific and helpful
  • Autocomplete attributes for common fields
  • Submit button clearly labeled

📱 Responsive Design

Breakpoints

  • Mobile (320px-767px): Single column, stacked elements
  • Tablet (768px-1023px): 2-column layouts
  • Desktop (1024px+): Full layout
  • Tested on all breakpoints (xs, sm, md, lg, xl, 2xl)

Mobile Optimization

  • Touch targets larger on mobile (min 44px)
  • Text readable without zoom (min 16px)
  • No horizontal scrolling
  • Hover states have touch alternatives
  • Animations reduced on mobile (or disabled)

Images

  • Responsive sizing (max-w-full h-auto)
  • Proper srcset for different resolutions
  • Lazy loading on below-fold images
  • Alt text provided

⚡ Performance

Animations

  • Use transform and opacity only (GPU-accelerated)
  • Duration 200-300ms for micro-interactions
  • Respects reduced motion preference (prefers-reduced-motion)
  • No layout shift during animations
  • Smooth 60fps on all devices

Assets

  • Images optimized (WebP format, compressed)
  • Icons use SVG (not PNG)
  • Fonts loaded efficiently (font-display: swap)
  • No render-blocking resources

Loading States

  • Skeleton screens for content loading
  • Spinners for actions < 3 seconds
  • Progress bars for known duration
  • Optimistic UI where appropriate

🔧 Code Quality

Component Structure

  • Props typed (TypeScript)
  • Default props defined
  • Variants clearly named (primary, secondary, ghost)
  • Size variants provided (sm, md, lg)
  • Disabled state implemented
  • Loading state implemented (if applicable)

Class Names

  • Uses Tailwind utilities (not custom CSS)
  • cn() utility for conditional classes
  • No hardcoded values (always use tokens)
  • Responsive modifiers where needed (sm:, md:, lg:)
  • Hover/focus/active states defined

Reusability

  • Component extracted if used 2+ times
  • Props allow customization without duplication
  • Composed from primitives where possible
  • No tight coupling to specific pages

🧪 Testing

Manual Testing

  • Visual regression (compare to design)
  • Keyboard navigation (tab through all elements)
  • Screen reader (VoiceOver/NVDA test)
  • Mobile device (test on real device)
  • Different browsers (Chrome, Firefox, Safari, Edge)

Automated Testing

  • Lighthouse accessibility score 100
  • axe DevTools no violations
  • Color contrast verified (WebAIM)
  • No console errors
  • Unit tests pass (if applicable)

📊 Component-Specific Checklists

Button Component

  • Border radius: rounded-2xl (primary), rounded-xl (secondary)
  • Min height: 44px (min-h-[44px])
  • Font: text-lg font-semibold (primary), text-base font-medium (secondary)
  • Padding: px-8 py-4 (primary), px-6 py-3 (secondary)
  • Gradient: Primary uses purple → blue → cyan
  • Shadow: Purple glow on hover (shadow-lg hover:shadow-xl)
  • Focus ring: ring-2 ring-purple-500/50
  • Hover: scale-105 + translate-y-1
  • Disabled state: opacity-50 cursor-not-allowed
  • Icon spacing: ml-2 or mr-2 (if icon present)

Card Component

  • Border radius: rounded-3xl (24px)
  • Background: white/5 backdrop-blur-md
  • Border: border border-white/10
  • Padding: p-8 (large cards), p-6 (medium cards)
  • Hover state: bg-white/8 border-white/20
  • Shadow: shadow-lg on hover (optional)
  • Transition: transition-all duration-300

Form Input

  • Border radius: rounded-xl (12px)
  • Min height: 44px (min-h-[44px])
  • Background: white/5 backdrop-blur-sm
  • Border: border border-white/10
  • Padding: px-4 py-3
  • Focus ring: ring-2 ring-purple-500/50
  • Label visible: Not placeholder-only
  • Error state: border-red-500 text-red-400

Pill/Badge

  • Border radius: rounded-full
  • Padding: px-4 py-2 (pill), px-3 py-1 (badge)
  • Font: text-sm font-semibold (pill), text-xs (badge)
  • Background: Gradient or solid semantic color
  • Icon spacing: gap-2 (if icon present)

🚨 Common Mistakes Reference

❌ AVOID THESE

// Wrong border radius
<button className="rounded-md">  // 6px too small

// Hardcoded colors
<div className="bg-[#8B5CF6]">  // Use tokens

// Missing touch target
<button className="px-3 py-1">  // Too small

// Placeholder-only label
<input placeholder="Email" />  // Need visible label

// Non-grid spacing
<div className="p-7">  // 28px breaks grid

// Missing focus state
<button className="...">  // Need focus:ring-2

// Missing aria-label on icon button
<button><X /></button>  // Need aria-label

// Hardcoded gradient
<div className="bg-gradient-to-r from-purple-600 to-cyan-600">  // Use preset

✅ DO THESE INSTEAD

// Correct border radius
<button className="rounded-2xl">  // 16px brand standard

// Use design tokens
<div className="bg-primary-600">  // Token-based

// Proper touch target
<button className="px-6 py-3 min-h-[44px]">  // WCAG compliant

// Visible label
<label htmlFor="email">Email</label>
<input id="email" placeholder="you@example.com" />

// Grid-aligned spacing
<div className="p-6">  // 24px (3× base)
<div className="p-8">  // 32px (4× base)

// Focus state included
<button className="... focus:outline-none focus:ring-2 focus:ring-purple-500/50">

// Aria label on icon button
<button aria-label="Close">
  <X className="h-5 w-5" aria-hidden="true" />
</button>

// Centralized gradient (future)
<div className={gradientPresets.buttonPrimary}>  // Use preset

📋 Pre-Merge Checklist Summary

Before merging your PR, confirm:

  1. Design Review

    • Matches design system tokens
    • Follows component patterns
    • No hardcoded values

  2. Accessibility Audit

    • Lighthouse score 100
    • Keyboard navigation works
    • Screen reader tested
    • Color contrast verified

  3. Responsive Check

    • Mobile tested (real device)
    • Tablet tested
    • Desktop tested
    • No horizontal scroll

  4. Performance Verified

    • No layout shift
    • Animations smooth
    • Images optimized
    • No console errors

  5. Code Quality

    • TypeScript types
    • Reusable component
    • Tests pass
    • PR reviewed

🔄 Review Process

1. Self-Review

Run through this checklist yourself before requesting review.

2. Peer Review

Another developer reviews code and functionality.

3. Design Review

Designer confirms visual accuracy and brand alignment.

4. Accessibility Review

Test with keyboard, screen reader, and automated tools.

5. Approval

All checkboxes must be ✅ before merging.
