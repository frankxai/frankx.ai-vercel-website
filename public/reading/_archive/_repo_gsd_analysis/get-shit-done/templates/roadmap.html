**Initial planning (v1.0):** - Phase count depends on depth setting (quick: 3-5, standard: 5-8, comprehensive: 8-12) - Each phase delivers something coherent - Phases can have 1+ plans (split if >3 tasks or multiple subsystems) - Plans use naming: {phase}-{plan}-PLAN.md (e.g., 01-02-PLAN.md) - No time estimates (this isn't enterprise PM) - Progress table updated by execute workflow - Plan count can be "TBD" initially, refined during planning

Success criteria:

  • 2-5 observable behaviors per phase (from user's perspective)
  • Cross-checked against requirements during roadmap creation
  • Flow downstream to must_haves in plan-phase
  • Verified by verify-phase after execution
  • Format: "User can [action]" or "[Thing] works/exists"

After milestones ship:

  • Collapse completed milestones in <details> tags
  • Add new milestone sections for upcoming work
  • Keep continuous phase numbering (never restart at 01)
- `Not started` - Haven't begun - `In progress` - Currently working - `Complete` - Done (add completion date) - `Deferred` - Pushed to later (with reason)
